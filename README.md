# Sgr A* Simulation

Interactive recreation of Andrew Chael's *Sgr A\* Simulation* visualization —
a GRMHD-style split-view of an accretion flow around Sagittarius A\*, rendered
in real time in the browser with custom GLSL shaders and React Three Fiber.

![Demo](public/grmhd/demo.gif)

## What you're looking at

The plot mirrors Andrew Chael's reference video. It's split into three panels by
two draggable separators:

| Quadrant                         | Field                       | Colormap            |
| -------------------------------- | --------------------------- | ------------------- |
| top-left  `x < uXSplit, y > uYSplit` | Radiation Power             | viridis             |
| bottom-left `x < uXSplit, y < uYSplit` | Magnetic Field Strength     | inferno             |
| right     `x > uXSplit`          | Density                     | NASA black→orange→yellow |

Plus:

- **Black-hole shadow** at the center, with the curved photon-ring lensing
  visible as a dark arc cutting through the orange plasma — this is *real* GR
  light-bending, baked into the data by the BHOSS ray-trace.
- **White iso-contour lines** on the left two quadrants. These are *derived
  from the data* (lines of constant Inu, drawn in the shader with `fwidth()`
  for constant screen-pixel width), so they trace the actual bright filaments
  instead of an unrelated procedural topology.
- **Three colorbars** on the left edge showing the radiation / density /
  magnetic-field tick magnitudes.
- A **scrubber + play/pause** at the bottom for the 3-minute loop.

## Two modes

A small pill in the top-right toggles:

- **Real GRMHD** *(default)* — left and right panels both sample a packed
  atlas of 101 ray-traced radiation-intensity frames from the *Yoon+ 2020*
  Sgr A\* GRMHD simulation library. The bright vertical streak inside the
  shadow is the actual jet from the simulation; the brightness pulsation is
  the actual MAD magnetic-flux eruption cycle.
- **Procedural** — everything is generated by GLSL noise + an analytic
  stream function. Useful as a comparison baseline against the real data.

## Draggable separators

Two thin bars with diamond grips sit at the panel boundaries:

- The **vertical bar** drags horizontally to slide the Density / Radiation+|B|
  split anywhere from −60 to +60 r_g.
- The **horizontal bar** drags vertically to slide the Radiation Power /
  Magnetic Field Strength split (left panel only).
- The **crosshair button** (top-right, next to camera reset) snaps both
  separators back to (0, 0).

Both splits live in `useRef`s, so dragging never re-renders the React tree —
the shader uniforms are updated each frame inside `useFrame`.

## Camera

OrbitControls: drag to orbit, right-drag to pan, scroll to zoom. The
**rotate-counterclockwise button** (top-right) resets the camera to the
default front-on view.

## Running

```bash
npm install
npm run dev
# open http://localhost:3000 (or PORT=3010 npm run dev if 3000 is taken)
```

On first load, a "tracing magnetic field lines…" bar appears for ~1 second
while the streamline worker precomputes 60 frames of line geometry. After
that the simulation runs at vsync.

## Real GRMHD data

`public/grmhd/` ships three pre-baked artifacts that the shader loads at
runtime:

- `inu_atlas.png` (~1.4 MB, LFS) — 101-frame atlas of edge-on ray-traced
  radiation intensity at 256×128 per tile, **sqrt-clip normalized** so the
  ~50% frame-to-frame brightness variation of the MAD pulsation stays
  visible. Laid out in an 11×10 grid.
- `radial_profiles.json` (~1.9 MB) — 120-frame time series of
  radially-resolved density / B² / synchrotron emissivity profiles for the
  developed-MAD regime.
- `manifest.json` — atlas layout, normalization ranges, and plot domain.

These were generated by `scripts/grmhd-to-textures.py` from two public
CC-BY-4.0 datasets:

- **Yoon, D., Chatterjee, K., Markoff, S., et al. (2020).** *"Spectral and
  Imaging properties of Sgr A\* from High-Resolution 3D GRMHD Simulations
  with Radiative Cooling."* MNRAS 499, 3178. arXiv:2009.14227.
  Data: [Zenodo 3988208](https://zenodo.org/records/3988208).
- **Sosapanta-Salas, L. D., Liska, M., Markoff, S., et al. (2025).**
  *"Two-temperature treatments in magnetically arrested disk GRMHD
  simulations more accurately predict light curves of Sagittarius A\*."*
  MNRAS, [staf240](https://doi.org/10.1093/mnras/staf240).
  Data: [Zenodo 14793884](https://zenodo.org/records/14793884).

To regenerate the textures (after downloading the source NPZs):

```bash
python3 -m venv venv && source venv/bin/activate
pip install numpy pillow
# download the two zips from the Zenodo links above, unzip them somewhere
python scripts/grmhd-to-textures.py \
  --yoon /tmp/yoon --salas /tmp/salas --out public/grmhd
```

To regenerate `public/grmhd/demo.gif` (requires `ffmpeg` and the dev server
running on :3010):

```bash
node scripts/capture-gif.mjs --duration 6 --fps 10 --out public/grmhd/demo.gif
```

## What's procedural and why

The right (Density) panel and left two panels all sample the **same**
real Yoon Inu atlas. The viridis/inferno/orange colormaps then interpret
that scalar field as Radiation Power, |B|, and Density respectively —
a defensible approximation because synchrotron emissivity is roughly
`ρ · |B|² · T_e`, so the three quantities share strong spatial
correlation.

The white field lines on the left are **shader-drawn iso-contours of the
real Inu field** (`isoContours(v, 0.075)` with `fwidth()` for constant
screen-space line width). They trace the actual bright filaments by
construction.

The blue/white streamlines visible in **Procedural mode** are integrated
through an analytic stream function (`shaders/lib/fields.ts`) and
precomputed in a worker. They're hidden in Real GRMHD mode because they
were designed for the procedural field topology and don't match the
real data's topology.

If a full 2D GRMHD snapshot dataset becomes available (density and B
field on a grid, not just ray-traced intensity), the texture-sampling
code in `shaders/simulation.frag.ts` and the worker in
`lib/streamlineWorker.ts` can be extended to consume it directly.

## Performance

A Web Worker precomputes every streamline frame at startup. Per-frame the
main thread just swaps which precomputed `BufferGeometry` the
`LineSegments` object renders — a single ref assignment. Simulation time
lives in a `useRef` and updates the shader uniform via `useFrame`, so the
React tree doesn't reconcile every animation frame (only the scrubber
readout updates at 4 Hz). Headless measurement: ~120 fps with stable
8.3 ms frame times.

The Inu atlas frames are sampled with **linear interpolation between
adjacent frames** (no smoothstep), so the plasma evolves with constant
change rate inside each ~1.8s per-frame window rather than pulsing at
frame boundaries.

## Tuning

The look lives in two places:

- `shaders/lib/fields.ts` — physics of density/B/radiation in Procedural
  mode. Change the `streamFunction` to reshape the field topology; tweak
  the `density` and `radiationPower` formulas for different emission
  patterns.
- `shaders/simulation.frag.ts` — colormaps and blends for Real GRMHD
  mode. The four-stop NASA-black-to-yellow ramp for the Density panel, the
  iso-contour spacing, and the per-quadrant gamma all live there.
